# Qlib 全 A 股选股研究平台设计

日期：2026-07-10  
状态：已完成对话设计确认，等待书面规格复核

## 1. 目标

在现有个人量化平台中建立一条可重复、可追踪的 Qlib 研究与每日选股闭环：

1. 从 Tushare 同步 2015 年以来的全 A 股日频数据。
2. 按交易日过滤 ST、停牌、新股、低流动性及无法正常成交的股票。
3. 使用 Qlib Alpha158 与 LightGBM 训练多持有周期模型。
4. 每日生成股票池、A-D 评级、动作信号和建议目标权重。
5. 在现有 React 界面中完成参数调整、任务重跑、模型审批、回测和结果查询。
6. 所有运行保留参数、数据、模型、状态、日志和耗时，结果可复现。

本阶段只生成研究结果、目标持仓与调仓建议，不自动向实盘模块下单。

## 2. 已确认的默认值

| 项目 | 默认值 |
| --- | --- |
| 股票范围 | 全 A 股动态历史股票集合 |
| 历史起点 | 2015-01-01 |
| 持有周期 | 1、5、10、20 个交易日，默认 5 日 |
| 模型 | Qlib Alpha158 + LightGBM |
| 自动运行 | 交易日 18:00 |
| 上市时间过滤 | 不足 120 个交易日剔除 |
| 流动性过滤 | 近 20 日平均成交额低于 5000 万元剔除 |
| 评级 | A 前 5%，B 5%-20%，C 20%-50%，D 后 50% |
| 每日股票池 | Top 50 |
| 买入集合 | Top 20 |
| 组合方式 | 默认等权，单票上限 5% |
| 调仓频率 | 与持有周期一致；默认每 5 个交易日一次性更换全部持仓 |
| 回测基准 | 沪深 300，可切换中证 500、中证 1000 |
| 佣金 | 双边万分之三 |
| 印花税 | 卖出万分之五 |
| 滑点 | 双边万分之五 |
| 本地存储 | SQLite + Parquet + Qlib 数据目录 |

以上过滤阈值、评级分位、Top N、持有周期、权重限制、基准和交易成本均可在策略参数页面修改。每次任务保存不可变的参数快照。

## 4. 总体架构

继续使用现有 React + TypeScript 前端和 FastAPI 后端，新增独立本地任务进程。

```text
React UI
  |
  | HTTP: 配置、任务创建、状态轮询、结果查询
  v
FastAPI
  |-- SQLite: 配置、任务、模型、信号、回测摘要
  |-- 只创建任务和读取结果，不执行长任务
  v
本地任务进程
  |-- Tushare -> Parquet 原始/中间数据
  |-- Parquet -> Qlib 本地数据目录
  |-- Qlib -> 模型文件、预测、回测产物
  `-- 成功后原子发布 SQLite 当前结果
```

FastAPI 不在请求线程中训练模型或同步全量数据。前端提交任务后立即获得运行 ID，通过状态接口轮询。任务进程串行执行重任务，避免个人电脑并发训练导致内存耗尽。

18:00 自动任务和手动任务调用相同的任务入口、参数校验和流水线，不维护第二套业务逻辑。Windows 任务计划只负责唤起命令，不包含业务规则。

## 5. 组件边界

### 5.1 API 层

职责：

- 校验请求和策略参数。
- 创建任务并返回运行 ID。
- 查询任务状态、日志摘要和产物。
- 查询股票池、信号、模型与回测结果。
- 发布或回滚生产模型。
- 不直接执行耗时任务。

### 5.2 任务运行器

职责：

- 从 SQLite 获取下一条排队任务。
- 获取任务类型锁，同类重任务串行执行。
- 更新任务阶段、进度、开始时间、结束时间和耗时。
- 把结构化日志写入任务记录和本地日志文件。
- 失败时保存错误摘要，成功时原子发布结果。

任务类型：

- `data_initialize`：2015 年以来历史初始化。
- `data_incremental`：补齐最近缺失交易日。
- `signal_run`：按参数快照重建股票池并生成信号。
- `model_train`：训练指定持有周期模型。
- `backtest_run`：回测指定模型与策略配置。
- `daily_pipeline`：增量数据、股票池、生产模型预测和评级的组合任务。

### 5.3 Tushare 数据适配器

职责：

- 读取本地 `.env` 中的 Token，不将 Token 传给前端或保存到数据库。
- 获取交易日历、股票基础信息、上市/退市信息、名称变更、日线、复权因子、停复牌和涨跌停价格。
- 按交易日或数据集断点续传。
- 对限频和临时网络错误进行有限次数重试。
- 对权限不足给出接口名、缺失能力及影响。

### 5.4 数据校验与 Qlib 转换

职责：

- 检查重复股票日期、交易日缺口、无效 OHLC、负成交量和缺失复权因子。
- 保留原始数据，不覆盖源文件。
- 生成前复权的 Qlib 特征数据和动态股票范围。
- 转换成功后切换当前 Qlib 数据版本，失败时继续使用上一版。

### 5.5 股票池服务

职责：

- 针对每个历史交易日建立当时可见的股票集合，避免幸存者偏差。
- 按顺序执行过滤规则并记录每一步数量。
- 为每只被剔除的股票记录一个或多个明确原因。
- 输出当日模型可评分集合和过滤漏斗。

### 5.6 Qlib 模型服务

职责：

- 为 1/5/10/20 日周期分别构建标签、训练、验证和测试。
- 使用 Alpha158 特征与 LightGBM。
- 保存模型、数据版本、特征版本、参数、随机种子和评价指标。
- 使用生产模型进行每日预测。
- 输出 LightGBM 单样本因子贡献，不额外部署 SHAP 服务。

### 5.7 评级与组合服务

职责：

- 在当日合格股票横截面内计算排名与分位。
- 按参数快照映射 A-D 评级。
- 生成买入、观察、持有/中性、回避动作。
- 生成 Top 20 等权目标，应用单票 5% 上限。
- 输出 Top 50 股票池、完整评级和结果说明。

### 5.8 回测服务

职责：

- 使用当时可交易股票池、指定模型、周期和参数进行时间序列回测。
- 信号在 T 日收盘后生成，从 T+1 开盘价进入，并在持有周期结束后的开盘价退出。
- 信号与评级每天更新，但组合只在持有周期到期时一次性更换全部持仓。
- 应用停牌、涨跌停、佣金、印花税和滑点约束。
- 输出净值、基准、回撤、月度收益、持仓、换手与交易成本。

## 6. 数据与股票池

### 6.1 数据落地

Parquet 按数据集和交易日期分区，支持幂等重跑。SQLite 只保存元数据和产品结果，不保存全量行情宽表。Qlib 数据放在独立目录，模型和回测产物按运行 ID 保存。

建议目录：

```text
backend/.data/
  quant.db
  raw/
  prepared/
  qlib/
  models/<model_id>/
  runs/<run_id>/
```

`backend/.data` 保持 Git 忽略。

### 6.2 动态股票集合

历史任一交易日只允许使用该日已上市且未退市的股票。不能用当前上市列表反推历史范围。名称与 ST 状态按当日有效信息判断。

### 6.3 默认过滤顺序

1. 当日有效上市状态。
2. 非 ST、*ST、退市整理。
3. 有有效行情且未停牌。
4. 已上市至少 120 个交易日。
5. 近 20 个交易日平均成交额至少 5000 万元。
6. 满足策略定义的可成交约束，不处于无法进入的涨跌停状态。

每次运行保存各步骤前后数量。过滤参数改变只影响新任务，不修改旧结果。

## 7. 标签、模型与防泄漏

### 7.1 标签口径

平台在 T 日收盘数据完整后产生信号。默认标签使用 T+1 开盘价进入，使用 T+h+1 开盘价退出，其中 `h` 为 1、5、10 或 20 个交易日的持有周期。进入日或退出日停牌、涨跌停且无法按开盘价成交的样本不进入训练标签。模型与回测使用相同价格口径。

特征只允许使用 T 日及之前的数据。股票池过滤也只允许使用当时已知数据。训练、验证和测试严格按时间顺序划分，不随机打乱。

### 7.2 训练与重训

- 每个持有周期训练独立模型。
- 首次训练使用 2015 年以来数据。
- 首次候选模型按交易日期使用 70% 训练、15% 验证、15% 样本外测试；测试集在调参和早停过程中不可见。
- 模型获批后，使用已经固定的特征和参数在截至标签完整日的全部可用数据上重训生产产物，并关联获批候选模型的样本外指标。
- 月度自动重训在每月首个交易日的每日流水线成功后运行，只创建候选模型，不自动发布。
- 用户可从界面手动训练指定周期与区间。

### 7.3 模型指标

- IC 与 Rank IC。
- Top 组合年化收益和相对基准超额收益。
- 最大回撤、夏普和换手率。
- 交易成本前后收益。
- 训练、预测和回测耗时。

### 7.4 模型生命周期

状态为 `candidate`、`production`、`archived`。每个持有周期最多有一个生产模型。发布新模型时原生产模型转为归档；回滚把指定归档模型重新设为生产。所有切换保留时间与来源模型，不删除历史产物。

## 8. 评级、动作与权重

每日对全部合格股票的预测分数降序排列：

- A：`[0%, 5%)`。
- B：`[5%, 20%)`。
- C：`[20%, 50%)`。
- D：`[50%, 100%]`。

默认 Top 50 构成每日股票池。Top 20 为买入候选并使用等权目标，单票不超过 5%；其余 B 为观察，C 为持有/中性，D 为回避。模型排名和建议每天更新，但真实目标组合只在对应持有周期到期时一次性调仓；默认 5 日周期即每 5 个交易日全量替换一次持仓。页面同时显示最近调仓日和预计下次调仓日。

每条信号至少包含：

- 交易日期、股票代码和名称。
- 持有周期、模型 ID 和数据版本。
- 原始预测分数、排名、分位和评级。
- 建议动作与目标权重。
- 相比上一交易日的排名和评级变化。
- 主要正向与负向贡献因子。
- 参数快照 ID 和任务运行 ID。

评级表示模型在当日横截面中的相对判断，不等同于收益保证。界面需显示此提示。

## 9. SQLite 核心实体

- `strategy_configs`：可编辑策略方案及版本。
- `task_runs`：任务类型、状态、阶段、进度、时间、耗时、参数快照、错误摘要。
- `data_versions`：原始与 Qlib 数据范围、校验状态和当前版本。
- `model_versions`：周期、状态、路径、训练区间、数据版本、参数和指标。
- `universe_runs`：交易日、参数快照、漏斗统计。
- `universe_members`：股票、是否合格、过滤原因。
- `signal_runs`：交易日、模型、周期、策略配置和运行状态。
- `stock_signals`：分数、排名、评级、动作、权重和贡献因子。
- `backtest_runs`：模型、区间、基准、成本、指标和产物路径。

旧的 `latest_signals.json` 在迁移完成后只保留兼容读取，不再作为真实结果主存储。

## 12. 任务状态与数据流

任务状态：`queued`、`running`、`succeeded`、`failed`、`interrupted`。

每日任务阶段：

```text
同步交易日与基础信息
  -> 同步当日行情和辅助数据
  -> 数据校验
  -> 更新 Qlib 数据
  -> 建立当日股票池
  -> 加载对应周期生产模型
  -> 预测与评级
  -> 组合目标权重
  -> 原子发布结果
```

手动重跑保存界面当时的参数快照。运行详情显示开始时间、结束时间和服务端计算的总耗时；前端不自行估算最终耗时。

## 13. 错误处理与恢复

- Tushare 临时失败：有限重试，超过次数后任务失败。
- Tushare 权限不足：显示接口和影响，不静默降级。
- 数据校验失败：不更新当前 Qlib 数据版本。
- 生产模型缺失：数据页面可用，信号任务失败并说明缺少哪个周期模型。
- 训练或回测失败：保留日志和已完成阶段，不改变生产模型。
- 服务在任务中退出：启动时把遗留 `running` 任务标记为 `interrupted`。
- 新结果失败：上一版股票池、信号和模型继续可查询。
- 前端空数据：显示初始化指引，不加载 mock 信号。
- Token 和本地敏感配置：不写日志、数据库、API 响应和前端。

第一版不实现任务取消和任意并发队列。个人本地串行任务已满足需求，避免引入额外进程协调复杂度。

## 15. 分阶段实施

### 阶段 1：数据与任务基础

- SQLite 模式和任务运行器。
- Tushare 全量/增量同步与 Parquet。
- 数据校验、动态股票池和过滤原因。
- 运行历史 API 与页面。

验收：小型固定数据和少量真实日期能够生成可查询股票池，任务状态与耗时正确。

### 阶段 2：Qlib 模型与信号

- Qlib 数据转换。
- 多周期标签、Alpha158 + LightGBM。
- 模型版本、预测、评级和权重。
- 模型发布与回滚 API。

验收：固定数据完成端到端闭环，生产模型能够生成每日信号。

### 阶段 3：研究工作台

- 今日股票池、评级与信号、策略参数。
- 模型训练与运行状态。
- 真实 K 线和右侧信号详情联动。

验收：前端不依赖 mock 信号，核心操作能从页面完成。

### 阶段 4：回测与自动运行

- 回测报告、模型对比和结果导出。
- 18:00 Windows 自动任务配置。
- 中文操作手册和最终真实数据验收。

验收：自动与手动任务使用同一流水线；回测、模型发布、每日信号形成完整闭环。

## 16. 实施约束

- 保留并适配当前已暂存的 UI 改动，不覆盖或提交无关用户修改。
- 沿用现有 FastAPI、React、TypeScript 和本地组件风格。
- 只为 Qlib 与 Parquet 增加必要依赖，不引入 Redis、Celery 或新的前端状态框架。
- 优先扩展现有领域对象和 API 模式，避免为了单一实现建立抽象工厂。
- 每个阶段先写能复现需求的测试，再实现最小通过代码。
- 每阶段结束必须运行后端测试和前端生产构建。
